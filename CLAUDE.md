# CLAUDE.md — meteo-guadalentin

## Project Overview

**Meteo Guadalentín** is a real-time weather monitoring and agricultural risk dashboard for the Guadalentín valley (Murcia, Spain), hosted at `meteoguadalentin.com` via GitHub Pages.

It aggregates data from ~91 personal weather stations (Weather Underground PWS) plus AEMET reference stations, calculates agricultural disease risk (DSV — Disease Severity Value, Gubler-Thomas formula), and serves an interactive map dashboard updated every 5 minutes automatically.

**No server, no database, no external Python dependencies.** Everything is Python stdlib + static JSON files + GitHub Actions.

---

## Architecture

```
GitHub Actions (cron: */5 * * * *)
        │
        ▼
check_historial.py  ──(fails)──►  fix_historial.py
        │(OK)
        ▼
mapa_totana.py
  ├─ Fetch ~91 WU stations (25 concurrent threads)
  ├─ gestionar_historial()         → history_24h.json   (24h rolling)
  ├─ gestionar_historial_agricola() → historial_agricola.json  (14-day)
  ├─ generar_json()                → data.json          (current snapshot)
  └─ generar_html()                → index.html         (full SPA, data embedded)
        │
        ▼
git commit + push → GitHub Pages serves static files
```

The SPA reads all data from the embedded JSON in `index.html` at load time. `data.json`, `history_24h.json`, and `historial_agricola.json` are also served as separate files (consumed by external tools or L&A Agrícola).

## Python Scripts Reference

| Script | When to run | Purpose |
|--------|-------------|---------|
| `mapa_totana.py` | Every 5 min (CI) or manually | Main: fetch all stations, update all JSON outputs, regenerate `index.html` |
| `check_historial.py` | Before main run (CI) | Validates `historial_agricola.json` has ≥50 temperature entries; exits 1 if not |
| `fix_historial.py` | When check fails | Downloads last 7 days from WU and rebuilds `historial_agricola.json` |
| `rellenar_historial.py` | Once (initial setup) | Backfills 7-day WU history across all stations |
| `dsv_desde_marzo.py` | Once per season | Calculates DSV accumulation from March 1st using WU owner API |
| `inicializar_dsv.py` | Once (alternative init) | Alternative: initialise DSV using AEMET climatological data |

### Execution in CI

```yaml
python3 check_historial.py || python3 fix_historial.py
python3 mapa_totana.py
git add -A && git commit -m "Meteo $(date)" && git push
```

### Manual local run

```bash
python3 mapa_totana.py            # full update cycle
python3 check_historial.py        # validate history (exit code: 0=OK, 1=needs fix)
python3 fix_historial.py          # repair corrupted or missing history
python3 rellenar_historial.py     # one-time 7-day backfill
python3 dsv_desde_marzo.py        # one-time seasonal DSV calc
```

## Data Files

### `data.json` — Current snapshot

```json
{
  "actualizado": "DD/MM/YYYY HH:MM:SS",
  "timestamp": "ISO8601",
  "estaciones": [
    {
      "id": "ITOTAN8", "nombre": "...", "lat": 37.xx, "lon": -1.xx,
      "temp": 22.5, "tempMin": 18.0, "tempMax": 28.0,
      "humedad": 55, "lluvia": 0.0, "lluviaHora": 0.0,
      "viento": 5.2, "rachaViento": 8.1, "presion": 1013.2,
      "dewPoint": 12.3, "uv": 4, "solar": 680, "actualizado": "..."
    }
  ]
}
```

### `history_24h.json` — 24-hour rolling history

```json
[
  { "timestamp": "ISO8601", "stations": [ /* same structure as WU observations */ ] }
]
```
Trimmed to the last 24 hours on every run. Entries older than 24h are dropped.

### `historial_agricola.json` — 14-day agricultural data

```json
{
  "YYYY-MM-DD": {
    "STATION_ID": {
      "tempMax": 30.1, "tempMin": 18.5,
      "precipTotal": 0.0, "humedadAltaMinutos": 90
    }
  }
}
```
`humedadAltaMinutos`: cumulative minutes where humidity ≥ 85% (increments 15 min per update). Used in DSV calculation. Retention: last 14 days (oldest deleted when count > 14).

### `historial_dsv.json` — Disease Severity Value

Cumulative DSV per station since the start of the season (March). Each entry tracks `dsv_acumulado`, `dsv_aemet`, `dsv_wu_extra`, `ref_estacion`, and dated records.

## GitHub Actions / CI

**Primary workflow:** `.github/workflows/actualizar.yml`

- **Trigger:** `*/5 * * * *` (every 5 minutes) + `workflow_dispatch`
- **Commit bot:** `meteo-bot@lya-agricola.com` / `Meteo Bot`
- **Commit message format:** `Meteo YYYY-MM-DD HH:MM`
- Both `git commit` and `git push` use `|| true` — failures are silently ignored to prevent blocking the cycle

Two additional workflow files exist at root level (`actualizar.yml`, `update-meteo.yml`) as 15-min and hourly variants. The canonical one is `.github/workflows/actualizar.yml`.

## API Keys & Secrets

| Key | Location | Notes |
|-----|----------|-------|
| `WUNDERGROUND_API_KEY` | GitHub Actions secret | WU API key; falls back to a hardcoded public key in `mapa_totana.py` if unset |
| `WU_KEY_OWNER` | Hardcoded in `dsv_desde_marzo.py` | Owner-tier key for historical 5-min data |
| AEMET JWT | Hardcoded in `inicializar_dsv.py` | Spanish met agency API token |

**Do not commit new secrets to source.** Use GitHub Actions secrets or environment variables. SSL certificate verification is intentionally disabled in all scripts (`ctx.check_hostname = False`, `ctx.verify_mode = ssl.CERT_NONE`) to work around WU API TLS quirks — do not "fix" this.

## Station Registry (`estaciones.txt`)

One WU station ID per line. Lines beginning with `#` are comments.

Station ID format: prefix + number (e.g., `ITOTAN8`, `ILORCA23`, `IALEDO5`, `IALHAM12`). The prefix encodes the municipality.

To add a station: append its WU station ID to `estaciones.txt`. It will be picked up on the next run of `mapa_totana.py`.

## Frontend (`index.html`)

- **Generated** by `mapa_totana.py::generar_html()` on every run — do not edit manually
- Single HTML file (~910KB) with all CSS, JS, and data embedded inline
- **Libraries (CDN):** Leaflet 1.9.4 (map), Turf.js 6 (IDW spatial interpolation for heatmaps)
- **Map center:** Totana, Murcia (37.76°N, -1.53°W)
- **Features:** parameter selector (temp/humidity/precipitation/wind), time-slider playback over `history_24h.json`, opacity slider, station pop-ups with custom names, grayscale tile option
- **Custom station names:** `nombresPersonalizados` object in the generated JS maps station IDs to friendly display names

To change the map layout, UI controls, or color scales: edit `generar_html()` in `mapa_totana.py`, then run the script to regenerate `index.html`.

## Coding Conventions

- **Language of code and comments:** Spanish (variable names, function names, print messages, comments)
- **Python style:** snake_case for functions and variables, UPPERCASE for module-level constants
- **No linter or formatter** — no Black, Flake8, mypy, or any config file
- **No external Python dependencies** — stdlib only (`urllib`, `json`, `ssl`, `os`, `datetime`, `concurrent.futures`, `zoneinfo`)
- **Error handling pattern:** `try/except Exception` returning `None` or empty defaults; API failures are printed and skipped, never crash the full run
- **Concurrency:** `ThreadPoolExecutor(max_workers=25)` for parallel WU station fetches
- **Timezone:** `zoneinfo.ZoneInfo("Europe/Madrid")` with fallback to `datetime.now()` for older Python

## Common Tasks

### Add a new weather station
1. Find the WU station ID (format: `I{MUNICIPALITY}{N}`)
2. Append to `estaciones.txt`
3. Run `python3 mapa_totana.py` — the station appears on the map immediately

### Repair corrupted history
```bash
python3 check_historial.py   # identify the problem
python3 fix_historial.py     # re-download 7-day history and rebuild
python3 mapa_totana.py       # regenerate all outputs
```

### Backfill history after downtime
```bash
python3 rellenar_historial.py   # downloads last 7 days for all stations
```

### Recalculate seasonal DSV
```bash
python3 dsv_desde_marzo.py      # recalculates from March 1st using WU owner API
```

### Test data integrity locally
```bash
python3 check_historial.py   # exit 0 = OK, exit 1 = insufficient data
python3 -c "import json; d=json.load(open('data.json')); print(len(d['estaciones']), 'stations')"
```
